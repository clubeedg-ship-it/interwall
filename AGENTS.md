## Project

**interwall**

interwall is a new standalone, enterprise-grade inventory management platform for small-to-medium manufacturing and assembly businesses. It replaces the existing `inventory-interwall` prototype and the separate `interwall-email-automation` service with a single multi-tenant system that owns inventory, orders, email ingestion, dashboards, and configuration in one codebase.

The product keeps the strongest parts of the original Interwall experience, especially the wall-style warehouse UI and visual design language, but rebuilds them on a modern React/Next.js and TypeScript stack with a proper backend, durable storage, tenant isolation, and maintainable business logic.

**Core Value:** Businesses can manage inventory, orders, kits, email-driven automation, and profitability in one durable multi-tenant system without the sync failures and architectural fragmentation of the prototype.

### Constraints

- **Architecture**: Build a single modern codebase rather than extending the split legacy systems — this is a core objective from the spec
- **Tenancy**: All tenant-scoped data must be isolated with row-level security and membership-aware access control
- **Frontend**: Use Next.js, React, TypeScript, Tailwind CSS, and shadcn/ui while preserving the recognizable Interwall visual identity
- **Backend**: Use Supabase or equivalent PostgreSQL-backed infrastructure with secure server-side functions for privileged operations
- **Business Logic**: FIFO valuation, kit consumption, reorder-point calculation, and email ingestion must be first-class backend concerns
- **Reuse**: Reuse existing Interwall UI patterns, barcode/FIFO logic, and email parsing heuristics where they help, but adapt them to the new schema and stack

## Frontend Architecture
- All JavaScript files are served directly to the browser without any build step
- Cache busting via manual query string versioning on script tags (e.g., `?v=21`)
- Versioned copies of files exist with timestamps (e.g., `app-1770129675.js`, `style-1770129675.css`)
- Global object pattern - each file exports a single object attached to `window`
- No ES modules, no CommonJS, no import/export statements in frontend code
- Cross-file communication via `window.state`, `window.CONFIG`, etc.
- In-memory global `state` object in `frontend/app.js` (lines 30-46)
- `Map` objects for O(1) lookups (`state.locations`, `state.parts`)
- localStorage persistence for user preferences and zone configuration

## Architecture

## Pattern Overview
- No frontend framework -- pure vanilla ES6+ JavaScript with Module Pattern (object literals exposed on `window`)
- InvenTree (Django/Python) used API-only as authoritative inventory backend
- Three-tier storage: localStorage (device-local fast cache), Config API (cross-device config sync), InvenTree API (authoritative inventory data)
- Docker Compose orchestration with nginx reverse proxy fronting everything
- Zero build step -- browser loads raw JS/CSS files directly
## Layers
- Purpose: All UI rendering, user interaction, barcode scanning
- Location: `frontend/`
- Contains: `index.html` (SPA shell with all view templates), `app.js` (core logic), `profit.js` (profitability engine), `tenant.js`, `labels.js`, `style.css`
- Depends on: InvenTree API, Config API, localStorage, Chart.js (CDN), Google Fonts (CDN)
- Used by: End users via browser (port 1441)
- Purpose: Serve static frontend files and reverse-proxy API requests to backend services
- Location: `frontend/nginx.conf`
- Contains: Proxy rules for `/api/` -> InvenTree, `/config-api/` -> Config API, `/media/` and `/static/` -> InvenTree
- Depends on: `inventree-server` container, `interwall-email-automation` container
- Used by: Frontend SPA (all API calls go through nginx)
- Purpose: Authoritative source for parts, stock items, locations, categories, sales orders
- Location: External Docker image `inventree/inventree:stable`
- Contains: Django REST API at `/api/`, PostgreSQL database, Celery background workers
- Depends on: PostgreSQL (`inventree-db`), Redis (`redis`)
- Used by: Frontend SPA via nginx proxy
- Purpose: Cross-device synchronization of fixed costs and fixed components configuration
- Location: Referenced as `interwall-email-automation` in `frontend/nginx.conf` (not in this repo's docker-compose)
- Contains: Express.js server with GET/POST `/api/config`, stores data as JSON file on disk
- Depends on: Filesystem storage
- Used by: Frontend SPA for syncing `fixed_costs` and `fixed_components`
- Purpose: Fast offline-capable cache, device-specific preferences, zone/shelf configuration
- Location: Browser localStorage (no server component)
- Contains: Auth token, zone config, shelf config, transactions (partially synced), theme, view state, inventory cache
- Depends on: Nothing
- Used by: All frontend modules

- Location: `frontend/app.js` (lines ~112-400)

## Entry Points
- Location: `frontend/index.html`
- Triggers: Page load in browser
- Responsibilities: Loads all CSS/JS, defines all HTML view templates and modals, bootstraps via `DOMContentLoaded` -> `init()` -> `auth.init()`
- Location: `docker-compose.yml`
- Triggers: `docker compose up -d`
- Responsibilities: Starts PostgreSQL, Redis, InvenTree server, InvenTree worker, Frontend nginx
- Location: `frontend/entrypoint.sh`
- Triggers: Docker container start
- Responsibilities: Generates `env.js` from environment variables (`API_URL`, `CONFIG_API_URL`), then starts nginx
## Error Handling
- All `api.request()` calls wrapped in try-catch, errors shown via `toast.show(message, 'error')`
- `notifications.show()` for persistent warnings (e.g., duplicate parts detection)
- Console logging with emoji prefixes for developer debugging
- No global error boundary or error reporting service
- Auth errors (401) fall through to token validation failure -> redirect to login

## Key Design Decisions
- **No frontend framework** -- intentional choice for zero build complexity, instant deploys, and maximum performance for a kiosk/warehouse environment
- **InvenTree as headless backend** -- leverages mature inventory management system without its default UI
- **Dual storage (localStorage + API)** -- enables offline-capable fast UX with eventual consistency to server
- **FIFO via Bin A/B naming convention** -- warehouse bins named `{Zone}-{Col}-{Level}-A` and `{Zone}-{Col}-{Level}-B` encode FIFO rotation in location names
- **Config API as separate sidecar** -- decouples cross-device config sync from InvenTree to avoid custom Django plugins
- **Cache-busted filenames** -- production uses timestamped filenames (e.g., `app-1770129675.js`) instead of query-string versioning
- **Nginx reverse proxy** -- all API traffic goes through frontend nginx, avoiding CORS issues and enabling single-origin deployment
