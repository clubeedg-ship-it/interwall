## Issue 1: Different Sales Data on Different Devices (IMPROVED)

### Problem
Transactions stored in localStorage are device-local. Different devices showed different sales history.

### Root Cause
- `recordSale.loadTransactions()` loads from localStorage first
- `fetchInvenTreeSalesOrders()` only added NEW orders (skipped if orderId exists)
- If Device A had local transactions, Device B wouldn't see them
- If both devices had the same orderId but different data, stale data persisted

### Solution
1. **Merge Strategy**: InvenTree is now authoritative for synced orders
   - If a local transaction has a matching InvenTree SO, update it with InvenTree data
   - This ensures cross-device consistency
   
2. **Recalculate Margin**: After merge, recalculate `totalMargin` from all transactions
   - Prevents drift from repeated loads/merges
   
3. **Save After Merge**: Save merged state to localStorage
   - Ensures local cache reflects server truth

### Files Changed
- `frontend/profit.js`:
  - Updated `profitEngine.init()` to recalculate margin after merge
  - Updated `fetchInvenTreeSalesOrders()` to update existing transactions
  - Changed skip logic to detect if update needed

### Testing
1. Create a sale on Device A
2. Wait for sync to InvenTree (check console for "Created Sales Order")
3. Open Device B, navigate to Profitability
4. **Expected**: Same sale appears on Device B with matching data

## Issue 2: Duplicate Components in Dropdowns (FIXED)

### Problem
Component dropdowns could show confusing options if parts had duplicate names.

### Root Cause
- Dropdown displayed `"Part Name (X in stock)"` format
- If two parts had the same name (different PKs/SKUs), users couldn't distinguish them

### Solution
Added SKU/IPN display in component dropdowns:
- For duplicate names: `"[SKU-001] Part Name (X in stock)"` - SKU first for visibility
- For unique names: `"Part Name (SKU-001) - X in stock"` - SKU after name

Added duplicate detection:
- Builds name count map before rendering dropdown
- Warns in console if duplicates found
- Shows toast warning once per session on app load

### Files Changed
- `frontend/profit.js`:
  - `fixedComponentsEditor.populatePartSelect()` - Added SKU display + duplicate detection
  - `recordSale.populatePartsDropdown()` - Added SKU display + duplicate detection
  
- `frontend/app.js`:
  - Added `checkDuplicateParts()` function
  - Called after `loadParts()` completes
  - Shows warning notification if duplicates found

### Testing
1. Create two parts with the same name in InvenTree
2. Open Record Sale modal
3. **Expected**: Dropdown shows `"[SKU-A] Part Name"` and `"[SKU-B] Part Name"`
4. Console shows warning about duplicate names

## Issue 3: Wrong Components Retrieved (MITIGATED)

### Problem
Users might accidentally select the wrong component if parts have identical names.

### Root Cause
- Component selection uses partId (correct implementation)
- But if user selects wrong part from confusing dropdown, wrong stock consumed
- No way to distinguish parts with same name

### Solution
This is now mitigated by Issue 2's fix:
- SKU/IPN shown in dropdown makes selection unambiguous
- Duplicate warning alerts users to fix data in InvenTree

### Recommendation
If duplicate parts exist in InvenTree:
1. Rename one part to be more specific
2. Or merge parts if they're actually the same product
3. Check InvenTree for parts with same name: `GET /api/part/?search=<name>`

## Remaining Known Issues

### Not Fixed (Architectural Limitations)

1. **Zone Config Not Synced**
   - Zone layout stored in localStorage only
   - Different devices can have different zone layouts
   - **Workaround**: Manually configure zones on each device
   - **Future Fix**: Move zone config to Config API

2. **Shelf Config Not Synced**
   - Per-shelf FIFO settings (split mode) are device-local
   - **Workaround**: Configure on each device
   - **Future Fix**: Move to Config API or InvenTree custom fields

3. **JIT Config Not Synced**
   - Reorder point settings stored locally
   - **Workaround**: Configure on each device

### Recommended Long-term Improvements

1. Make InvenTree Sales Orders the SOLE source of truth for transactions
   - Remove localStorage for transactions entirely
   - All devices read from InvenTree

2. Move zone/shelf/JIT config to Config API
   - Single source of truth for all configuration

3. Add real-time sync (WebSocket or polling)
   - Instant updates across devices
