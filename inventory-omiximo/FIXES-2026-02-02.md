# omiXimo Inventory Fixes - 2026-02-02

## Summary
Fixed sync issues and added sale editing capability.

---

## Issue 1: Fixed Components Not Syncing (FIXED)

### Problem
- Frontend saves without `sku` field → **Already fixed in prior commit**
- When values change, other devices don't see updates (localStorage issue)

### Root Cause
The `costConfig.init()` and `fixedComponentsConfig.init()` methods were:
1. Loading from localStorage FIRST
2. Then async loading from backend (too late - UI already rendered with old data)

This meant Device A's changes wouldn't be seen by Device B until Device B manually refreshed AND the async backend call completed.

### Solution
Made backend the **source of truth**:
- `init()` now **awaits** `loadFromBackend()` FIRST
- Only falls back to localStorage if backend is unavailable
- This ensures all devices see the same config on page load

### Files Changed
- `frontend/profit.js`:
  - `costConfig.init()` - Now async, awaits backend load
  - `costConfig.loadFromBackend()` - Returns boolean for success
  - `costConfig.loadFromLocalStorage()` - New method (renamed from `load()`)
  - `fixedComponentsConfig.init()` - Same pattern as costConfig
  - `fixedComponentsConfig.loadFromBackend()` - Returns boolean
  - `fixedComponentsConfig.loadFromLocalStorage()` - New method
  - `profitEngine.init()` - Now awaits both config inits

## Issue 3: Sale Editing Feature (ADDED)

### Problem
No ability to edit a sale after it was recorded. If automation made a mistake, the only option was to delete and re-create.

### Solution
Added edit capability to the Record Sale modal:

1. **Edit Button**: Added edit icon button next to delete button on each transaction card
2. **Edit Mode**: Modal now supports editing existing sales
3. **Smart Stock Handling**: 
   - Existing components marked with `isEdit: true` flag
   - Only NEW components consume stock on save
   - Original sale date preserved on edit
4. **Visual Feedback**: Modal title shows "Edit Sale: ORD-XXX" when editing

### Files Changed
- `frontend/profit.js`:
  - Added `currentEditOrderId` property to `recordSale`
  - Added `showEdit(orderId)` method
  - Updated `show()` to reset edit mode
  - Updated `hide()` to clear edit mode
  - Updated `submit()` to handle edit vs new sale
  - Updated `renderTransactions()` to include edit button
- `frontend/style.css`:
  - Added `.transaction-actions` container styles
  - Added `.transaction-edit-btn` styles (matches delete button pattern)

### Limitations
- **InvenTree orders cannot be edited** - User gets a warning message and is directed to edit in InvenTree directly
- **Component changes on edit don't restore original stock** - If you remove a component during edit, that stock isn't returned. Use delete + re-create for major changes.

## Architecture Notes

### Backend Config Flow (After Fix)
```
Page Load
    ↓
profitEngine.init()
    ↓
await costConfig.init()
    ↓
await loadFromBackend() ──→ Success? ──→ Use backend data
    ↓ (fail)                              ↓
loadFromLocalStorage()                 Cache to localStorage
```

### Edit Mode Flow
```
Click Edit Button
    ↓
showEdit(orderId)
    ↓
Find transaction
    ↓
Set currentEditOrderId
    ↓
Populate form with existing data
    ↓
Mark existing components as isEdit: true
    ↓
User makes changes
    ↓
submit()
    ↓
Skip stock consumption for isEdit components
    ↓
Update transaction in place (preserve date)
    ↓
Recalculate totalMargin
```
